# FastAPI Monorepo Backend Design

## Goal

Reorganize the project into a monorepo and introduce a separate FastAPI backend. The Next.js app should become the frontend layer, while FastAPI gradually takes ownership of authentication, project data, media, uploads, database access, and storage integration.

## Current State

The repository is currently a single Next.js application. It already contains backend responsibilities inside the Next app:

- Server Actions for admin mutations.
- Route handlers under `src/app/api` for media and uploads.
- Postgres persistence through Drizzle.
- S3/MinIO storage helpers.
- Admin authentication through a session cookie.
- Docker Compose services for the web app, Postgres, MinIO, and MinIO setup.

The migration should preserve the working app while responsibilities move to FastAPI.

## Approved Architecture

Use a monorepo with two applications:

- `apps/web`: the existing Next.js frontend and admin UI.
- `apps/backend`: a new FastAPI service.

The FastAPI service will use plain `pip` with `requirements.txt`. The first backend version starts simple, then becomes the system of record for API behavior.

Avoid introducing shared packages until there is a concrete need. API contracts can start as OpenAPI schemas generated by FastAPI and TypeScript types can be added later if duplication becomes painful.

## Authentication

Move admin authentication to JWT.

The target flow is:

- `POST /auth/login` receives admin credentials.
- FastAPI verifies the credentials against the admin user data.
- FastAPI returns an access token.
- The frontend sends authenticated API requests with `Authorization: Bearer <token>`.

The current cookie/session flow can remain during migration only where needed to keep existing admin screens working. The final state should not depend on Next.js Server Actions or Next-managed admin sessions.

## Backend Responsibilities

FastAPI should own these areas after migration:

- Admin login and JWT verification.
- Published project reads for public pages.
- Admin project CRUD.
- Project section CRUD.
- Media asset creation and listing.
- S3/MinIO upload signing and delivery URL behavior.
- Video processing with FFmpeg.
- Database access and validation for project/media rules.

Next.js should keep these responsibilities:

- Public portfolio rendering.
- Admin UI rendering.
- Client-side form behavior and API calls.
- Frontend routing and visual experience.

## Migration Strategy

Use an incremental migration instead of a big-bang rewrite.

### Phase 1: Monorepo Structure

Move the current Next app into `apps/web`. Update root scripts, TypeScript paths, Docker files, and documentation so the app still runs as before.

Create `apps/backend` with a minimal FastAPI app, `requirements.txt`, config loading, and `/health`.

### Phase 2: Backend Foundation

Add backend environment configuration, CORS for the frontend origin, basic error response shape, and JWT auth endpoints.

The backend should be independently runnable in local development.

### Phase 3: Data API Migration

Expose project, section, and media endpoints from FastAPI. Update the Next frontend/admin to read and write through HTTP API calls instead of direct Server Actions for migrated flows.

### Phase 4: Upload and Media Migration

Move upload signing, media object verification, and video processing to FastAPI. Preserve the current S3/MinIO behavior and FFmpeg output expectations.

### Phase 5: Remove Old Backend Code From Next

After all migrated flows are verified, remove obsolete Next route handlers, Server Actions, and server-only database/storage helpers from `apps/web`.

## Database and Migrations

The current database is Postgres. The migration should preserve existing data.

For the first backend phase, do not change the schema unless required by the API boundary. Database migration tooling can be decided during implementation planning. The likely options are:

- Keep existing Drizzle migrations temporarily while FastAPI is introduced.
- Move to Python-native migrations later if the backend becomes the only database owner.

The implementation plan must choose one explicit short-term migration owner before data endpoints are moved.

## Docker and Local Development

Docker Compose should eventually run:

- `web`: Next.js frontend.
- `backend`: FastAPI API.
- `postgres`: existing database.
- `minio`: existing S3-compatible storage.
- `minio-init`: existing bucket/CORS setup.

Local development should support running each app separately:

- Web app from `apps/web`.
- Backend app from `apps/backend`.

Root scripts should provide convenient commands for both.

## Error Handling

FastAPI should return predictable JSON errors for validation and auth failures. The frontend should display user-facing Portuguese messages in the admin UI, preserving the current UX where possible.

Internal errors should not leak secrets or stack traces to clients.

## Testing and Verification

At minimum, each phase should keep these checks passing:

- Next.js lint/build for the web app.
- FastAPI import/startup check for the backend.
- `/health` returns success for the backend.
- Existing public project page still renders after monorepo migration.
- Admin login and project editing continue to work during migration.

Later phases should add backend tests for auth, project CRUD, and upload validation.

## Out Of Scope

- Replacing the frontend design or admin UX.
- Adding multi-user roles beyond the current admin concept.
- Switching away from Postgres.
- Changing S3/MinIO provider behavior.
- Introducing shared packages before they are needed.
- Fully rewriting all backend behavior in the first implementation step.

## Success Criteria

- The repository is organized as a monorepo with `apps/web` and `apps/backend`.
- FastAPI runs independently and exposes `/health`.
- The current app remains functional after the initial structure change.
- JWT is the target authentication model for migrated admin API calls.
- There is a clear path to move all backend responsibilities from Next.js to FastAPI without a big-bang rewrite.
